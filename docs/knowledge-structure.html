Knowledge Organization

Last Updated: April 8, 2025

The Integrated Code Rules and Memory Bank System uses a tiered knowledge structure to optimize the loading and management of project information. This approach ensures that only the most relevant information is loaded for each task, minimizing token usage while maintaining project continuity.

Key Principle: Information is organized in four tiers with task-oriented loading priorities, from minimal required knowledge to detailed reference information.

Tiered Knowledge Structure

Knowledge is organized in four tiers with progressively increasing detail and decreasing loading priority:

Bootstrap Tier

Core system structure, loaded only when essential for understanding the command system.

Critical Tier

Current state relevant to immediate tasks, loaded only for files directly relevant to current task step.

Essential Tier

Reference information loaded only when task requirements aren't clear from Critical tier.

Reference Tier

Detailed implementation information accessed only when directly relevant to current task step.

Tier Details

1. Bootstrap Tier (Minimal Required Knowledge)

  • bootstrap.md - Core system structure, loaded only when essential
  • Access only when needed to understand command system
  • Contains fundamental information about system operation
When to Access Bootstrap Tier

Only access the Bootstrap tier when:

  • Setting up the system for the first time
  • Needing to understand core command structure
  • Troubleshooting fundamental system issues

2. Critical Tier (Task-Relevant Only)

  • activeContext.md - Current state relevant to immediate task
  • progress.md - Status information needed for current step
  • session_cache.md - Minimal continuity information if continuing a task
  • Load only files directly relevant to current task step
Critical Tier Loading

The Critical tier is your primary source of task-relevant information. Always start here, loading only the specific files needed for your current step.

3. Essential Tier (Load Only When Required)

  • projectbrief.md - Reference only when task scope is unclear
  • .cursorrules - Reference only when implementation patterns are needed
  • Load only when task requirements aren't clear from Critical tier
Essential Tier Loading

Only move to Essential tier files when you need clarity on project scope or implementation patterns that aren't available in the Critical tier.

4. Reference Tier (Avoid Unless Specifically Needed)

  • productContext.md - Why and how the project works
  • systemPatterns.md - Architecture and design patterns
  • techContext.md - Technical implementation details
  • Load only specific files when directly relevant to current task step
Reference Tier Loading

Reference tier files contain detailed information that should only be loaded when specifically needed for a task step. Avoid loading these files "just in case."

Task-First Loading Process

The knowledge tiering system supports a task-first approach to information loading:

  1. Analyze the immediate task requirements
    • Break down the task into discrete steps
    • Identify what information is needed for the current step only
  2. Identify the minimal set of files needed
    • Start with Critical tier files relevant to the task
    • Only add Essential or Reference tier files if specifically needed
  3. Load only those files directly relevant
    • Use read_mb [file] to load specific files
    • Avoid loading entire directories or tiers
  4. Execute the current step completely
    • Focus on completing the current step with the loaded information
  5. Load additional files only when needed for the next step
    • Evaluate what's needed for the next specific step
    • Load only those additional files
  6. Update only files with meaningful changes
    • Follow the Documentation Decision Framework
    • Make targeted updates to relevant files

Documentation Decision Framework

The system uses a standardized framework to determine which files to update based on the type of change:

Change Type Documentation Requirements
Interface changes Update API docs, activeContext.md
Implementation details Code comments only
Architecture changes Update systemPatterns.md
New features Update progress.md, projectbrief.md
Bug fixes Update progress.md only
Refactoring Minimal documentation unless patterns change

File Purposes and Content

Bootstrap Tier

bootstrap.md

Core system structure and fundamental operation information.

  • Basic command structure
  • System initialization process
  • Core operational principles

Critical Tier

activeContext.md

Current state relevant to immediate tasks.

  • List of active tasks with current status
  • Components currently being modified
  • Decisions directly affecting current tasks
  • Specific next steps for current tasks

progress.md

Status tracking information needed for current steps.

  • Task progress tracking
  • Completed milestones
  • Current blockers or issues
  • Immediate next actions

session_cache.md

Minimal continuity information for continuing tasks.

  • Session status (continuing or complete)
  • Current task in progress
  • Current step in the process
  • Critical files needed to continue
  • Minimal state information needed to continue

Essential Tier

projectbrief.md

Core requirements and project scope information.

  • Project goals and objectives
  • Key requirements
  • Main features and functionality
  • Project constraints and limitations

.cursorrules

Project patterns and implementation guidelines.

  • Coding patterns and standards
  • Implementation guidelines
  • Code organization principles
  • Best practices specific to the project

Reference Tier

productContext.md

Why and how the project works from a product perspective.

  • Detailed product description
  • User personas and use cases
  • Product flows and interactions
  • Business requirements and constraints

systemPatterns.md

Architecture and design patterns used in the project.

  • System architecture overview
  • Design patterns and their implementations
  • Component interactions and dependencies
  • Data flow and state management

techContext.md

Technical implementation details and technology stack.

  • Technology stack details
  • Library and framework usage
  • API specifications and integrations
  • Technical constraints and considerations

Loading Commands and Examples

Basic Loading Commands

read_mb                  # Load Critical tier files needed for current task
read_mb [file]          # Load specific file only
read_mb standard        # Load Critical + Essential tiers
read_mb complete        # Load all Memory Bank files (rarely needed)
update_mb [file]        # Update specific file with minimal changes

Example: Task-First Loading Process

Example: Implementing Form Validation

Step 1: Analyze Task Needs

Need to understand current form handling and validation requirements.

do_task implement form validation
read_mb activeContext.md        # Current state of form components
read_mb progress.md             # Track validation task progress

Step 2: Implement First Component

Need implementation patterns for validation logic.

read_mb .cursorrules            # Get implementation patterns

Step 3: Need Technical Details

Need specific technical information about the validation library.

read_mb techContext.md          # Technical implementation details

Step 4: Update Documentation

update_mb activeContext.md      # Update form validation status
update_mb progress.md           # Mark validation implementation as complete

Best Practices for Knowledge Organization

  1. Prioritize Critical Tier: Always start with Critical tier files relevant to your immediate task
  2. Be Specific in Loading: Load only specific files, not entire tiers or directories
  3. Progressive Loading: Add files only as needed for each step, not preemptively
  4. Targeted Updates: Update only files with meaningful changes related to your task
  5. Maintain Documentation Quality: Follow the Documentation Decision Framework for consistency
  6. Keep Session Cache Minimal: Include only essential information needed to continue tasks
  7. Use Templates: Follow established templates when creating or updating documentation

Common Mistakes to Avoid:

  • Loading all files "just to be safe"
  • Starting with Reference tier files instead of Critical tier
  • Making comprehensive documentation updates for minor changes
  • Including non-essential information in session_cache.md
  • Not following the Documentation Decision Framework
