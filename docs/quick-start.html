Quick Start Guide

Last Updated: April 8, 2025

This guide will help you get started with the Integrated Code Rules and Memory Bank System quickly. Follow these steps to begin using the system effectively.

Remember: The key to this system is loading only what you need when you need it. Don't try to load everything at once!

Step 1: Set Up Your Environment

Begin by setting up your environment with the necessary files:

  1. Clone or download the Memory Bank repository
  2. Ensure the core files are in place (especially the Bootstrap and Critical tier files)
  3. Verify that the Memory Bank directory structure follows the recommended pattern
memory-bank/
├── bootstrap.md
├── activeContext.md
├── progress.md
├── session_cache.md
├── projectbrief.md
├── .cursorrules
└── memory-bank/
    ├── productContext.md
    ├── systemPatterns.md
    └── techContext.md

Step 2: Starting Your First Session

To begin your first session with the system:

  1. Use the start_session command to initialize a new session with a timestamp
  2. Determine the specific task you want to accomplish
  3. Use the do_task [task] command to begin task execution
Example 
start_session
do_task implement new user authentication feature

Step 3: Loading Only What You Need

The system is designed to load only the minimal information needed for each task step:

  1. Start by analyzing what you need for the immediate task step
  2. Use read_mb to load only the Critical tier files needed
  3. Only load additional files if directly relevant to the current step
Example 
read_mb activeContext.md
read_mb progress.md

Key Principle: Avoid loading entire directories or unnecessary files. If you need information about system patterns, load systemPatterns.md only when you're ready to implement that specific part.

Step 4: Executing Task Steps

Execute your task following these principles:

  1. Complete the current step fully before looking ahead
  2. Use the standardized tool format for all operations
  3. Make surgical, precise changes when modifying files
  4. Verify changes against code standards

Example: Reading a File

<read_file>
<path>src/main.js</path>
</read_file>

Step 5: Managing Session State

When you need to continue a task in a future session:

  1. Use cache_session to create a continuation point
  2. Update only the necessary Memory Bank files with minimal changes
  3. In the next session, use continue_session to resume
Example 
cache_session

// Later, in a new session:
continue_session
read_mb session_cache.md

Step 6: Completing Tasks

When you've completed a task:

  1. Use complete_task to finish the current task
  2. Update the necessary documentation based on the Documentation Decision Framework
  3. Mark the session as complete with complete_session
Change Type Documentation To Update
Interface changes API docs, activeContext.md
Implementation details Code comments only
Architecture changes systemPatterns.md
New features progress.md, projectbrief.md

Common Commands Reference

Task Execution

  • do_task [task] - Execute specific task with minimal context loading
  • continue_task - Resume previous task using minimal context from cache
  • complete_task - Finish current task with targeted documentation updates

Memory Management

  • read_mb - Load Critical tier files needed for current task
  • read_mb [file] - Load specific file only
  • read_mb standard - Load Critical + Essential tiers
  • update_mb [file] - Update specific file with minimal changes

Session Management

  • continue_session - Flag this as a continuation; prioritize session_cache.md
  • complete_session - Mark session as complete, update necessary docs
  • cache_session - Create continuation point with minimal updates
  • start_session - Begin new session with fresh timestamp

Congratulations! You're now ready to start using the Integrated Code Rules and Memory Bank System. Remember the key principle: load only what you need when you need it.
